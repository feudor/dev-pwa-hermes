# PWA‑Hermes — требования MVP

## 1) Назначение

PWA‑Hermes — installable PWA для общения с Hermes вне Telegram: в браузере/на рабочем столе/на мобильном устройстве, с упором на голосовой сценарий и сохранением текстового канала.

## 2) MVP‑область

В MVP входят:

- до 5 параллельных чатов на пользователя;
- текстовые сообщения;
- голосовые сообщения через кнопку **«Записать аудио послание»**;
- STT: аудио пользователя превращается в текстовую расшифровку;
- Hermes core получает нормализованный текстовый вход;
- ответ ассистента показывается текстом;
- TTS: ответ ассистента доступен как аудио для прослушивания;
- обработка без стриминга: интерфейс ждёт завершения через polling;
- минимальные UI‑контуры разделов **Проекты**, **Планы**, **Cron**;
- серверное хранение истории, аудио и результатов обработки;
- TTL для исходных голосовых файлов: ориентир около 30 дней.

Не входит в MVP:

- realtime streaming токенов/аудио;
- полноценная ролевая модель;
- сложная криптография и offsite backup;
- marketplace/публичная регистрация пользователей;
- полноценный CRUD планировщика, если это тормозит голосовой контур.

## 3) Основные сценарии

### 3.1 Создать чат

1. Пользователь открывает раздел **Чаты**.
2. Нажимает **Новый чат**.
3. Сервер создаёт `chatId`.
4. UI открывает пустой чат.

Ограничение: если у пользователя уже 5 активных чатов, UI показывает понятное сообщение и не создаёт новый чат.

### 3.2 Отправить текст

1. Пользователь вводит текст в нижней панели чата.
2. UI отправляет `POST /api/chats/{chatId}/messages/text`.
3. Сервер создаёт сообщение и job.
4. UI poll’ит `GET /api/jobs/{jobId}`.
5. После `answered` в чате отображаются текст пользователя, ответ ассистента и, если TTS включён, `voiceReplyUrl`.

### 3.3 Отправить голос

1. Пользователь нажимает **Записать аудио послание**.
2. UI записывает аудио.
3. После остановки записи показывает preview и кнопку **Отправить**.
4. UI отправляет multipart `POST /api/chats/{chatId}/messages/voice`.
5. Сервер сохраняет оригинал аудио, создаёт job и запускает STT.
6. UI poll’ит `GET /api/jobs/{jobId}`.
7. После `answered` UI показывает:
   - расшифровку речи пользователя;
   - текст ответа ассистента;
   - кнопку play для TTS‑ответа.

### 3.4 Прослушать ответ

- Автовоспроизведение выключено.
- Пользователь сам нажимает play у голосового ответа.

## 4) Состояния job

Базовые статусы:

- `ingested` — вход принят сервером;
- `transcribing` — идёт STT;
- `ready` — вход нормализован, можно формировать ответ;
- `answering` — Hermes формирует ответ;
- `synthesizing` — формируется TTS‑аудио;
- `answered` — результат готов;
- `error` — обработка завершилась ошибкой.

UI обязан уметь показывать статус даже без точного процента прогресса.

## 5) Хранение и retention

Сервер хранит:

- chats;
- messages;
- jobs;
- текстовые transcript/assistantText;
- URL/путь к исходному аудио пользователя;
- URL/путь к TTS‑аудио ответа.

Retention:

- исходные голосовые файлы пользователя удаляются примерно через 30 дней;
- transcript и текст ответа остаются в истории, если пользователь явно не удалил чат;
- TTS‑ответы можно хранить отдельно с таким же или более коротким TTL, финальное правило уточняется позже.

## 6) UI‑принципы

- Только светлая тема.
- shadcn‑эстетика, grayscale base.
- Цвет — только как семантический статусный акцент.
- Русские UI‑тексты.
- Иерархия важнее плоских списков.
- Главная голосовая CTA: **«Записать аудио послание»**.

## 7) Критерии готовности MVP

MVP можно считать готовым, если:

- пользователь создаёт и переключает до 5 чатов;
- можно отправить текстовое сообщение;
- можно записать, preview’нуть и отправить голосовое сообщение;
- transcript появляется после STT;
- ответ ассистента появляется после обработки;
- голосовой ответ можно включить по клику;
- polling по `jobId` работает до `answered`/`error`;
- ошибки показываются понятным русским текстом;
- есть минимальная навигация: Чаты, Проекты, Планы, Cron;
- исходные аудио имеют retention‑политику.
