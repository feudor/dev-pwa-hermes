# MVP: PWA‑Hermes (Voice‑Only)

## 0) Цели MVP
- Пользователь общается с Hermes **через голос**.
- Пользователь управляет **до 5 параллельными чатами**.
- Ответ даётся **текстом + голосом (TTS)**.
- **MVP без стриминга**: результат появляется только после завершения обработки.

## 1) Основные пользовательские сценарии

### 1.1 Создать чат
- Пользователь открывает “Чаты”.
- Нажимает “Новый чат”.
- Получает `chatId` и пустой диалог.

### 1.2 Отправить голосовое сообщение
- В экране чата пользователь нажимает “Записать аудио послание”.
- Клиент записывает голос, формирует аудио‑файл.
- Клиент отправляет аудио на сервер.
- Клиент показывает статус: “обработка…”.
- После готовности:
  - в чат добавляется запись пользователя с `transcriptText`
  - в чат добавляется ответ ассистента с `assistantText`
  - добавляется аудио ответ (`voiceReplyUrl`) для прослушивания

### 1.3 Слушать голосовой ответ
- Пользователь нажимает play у голосового ответа.

### 1.4 Поведение при ошибках
- При `jobStatus=error` показываем ошибку + “повторить отправку” (если аудио сохранено на клиенте/есть повторный доступ).

## 2) Нефункциональные требования
- История/контент хранится на сервере.
- Голосовые файлы удаляются по TTL (~месяц, точное значение позже).
- Архитектура рассчитана на последующее подключение шифрования/политик приватности.

## 3) Продуктовые развилки (зафиксировано на MVP)
- Контекст/память: **уточним позже**.
- STT: **обязательно** (иначе невозможно дать ассистенту вход).
- TTS: **по умолчанию включён**.
- Streaming: **нет**.

## 4) Что считаем “готово”
- 1) UI поддерживает до 5 чатов.
- 2) Есть запись аудио и отправка.
- 3) Видим transcript.
- 4) Видим assistantText.
- 5) Есть play голосового ответа.
- 6) Polling по `jobId` до `answered`.
- 7) Cron/планировщик/проекты — минимальные экраны (UI‑контур) готовы (логика может быть упрощена).
