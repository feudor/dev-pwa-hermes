# PWA‑Hermes — UI Flow & Screens (MVP voice‑only)

> MVP: голос → STT → Hermes → TTS, **без стриминга**. Результат показывается после обработки (polling).
> Параллельно — до **5** чатов.

## 0) Общие принципы UI (для iPhone/iPad/macOS Safari)
- **Нет тёмной темы**.
- Монохромная shadcn-эстетика: нейтрали + **один** семантический акцент.
- Ошибки и статусы — функциональные (jobStatus), без “ярких красок”.
- Все тексты: русская UI-локаль.

## 1) Информационная архитектура MVP
### 1.1 Экраны
1. **Панель: Чаты**
   - список чатов (до 5 на пользователя)
   - кнопка “Новый чат”
   - бейдж статуса последнего job’а (если актуально)
2. **Экран: Чат**
   - поток сообщений
   - блок записи аудио
   - список/карточка последнего job’а (если в процессе)
3. **Экран: Планы / Cron / Проекты** (минимальный UI‑контур)
   - MVP допускает упрощённые формы: CRUD не полностью функционален, но навигация и структура должны быть.

### 1.2 Компоненты (уровень “страница”)
- `AppShell` (верхняя шапка/заголовок, навигация по разделам)
- `ChatList` (список чатов)
- `ChatThread` (сообщения + статусы)
- `VoiceRecorder` (кнопка записи/отправки)
- `MessageBubble` (user/assistant)
- `JobStatusPanel` (ингест/транскрипция/готово/ошибка)
- `VoiceReplyPlayer` (play voiceReplyUrl)

## 2) Экран “Чаты” (Chat List)
### 2.1 Входные данные
- `chats[]` (chatId, title, updatedAt)
- для каждого чата: `latestJobStatus` (опционально)

### 2.2 UI-секции
1. Заголовок: “Чаты”
2. CTA: “Новый чат”
3. Список:
   - каждая строка = карточка чата
   - элементы карточки:
     - title (или “Чат без названия”)
     - last activity (updatedAt)
     - (опционально) индикатор: processing/ready/error

### 2.3 Empty state
- Если чатов нет: подсказка “Создай первый чат” и CTA.

### 2.4 Ключевые действия
- Tap по карточке → переход в `ChatThread(chatId)`.
- Tap “Новый чат” → POST `/api/chats` → push/update локального state.

## 3) Экран “Чат” (Chat Thread)
### 3.1 Входные данные
- `messages[]` с полями из контракта (messageId, role, transcriptText, assistantText, voiceReplyUrl, jobStatus, inputType)

### 3.2 UI-секции
1. **Шапка чата**
   - title
   - (опционально) кнопка “Перейти в проект” если chat привязан к projectId (MVP: можно скрыть)
2. **Поток сообщений** (вертикальная лента)
   - `MessageBubble` user voice:
     - transcriptText (после ready)
     - если transcript ещё не готов:
       - “распознаю речь…” (jobStatus)
   - `MessageBubble` assistant:
     - assistantText
     - play voiceReplyUrl (если есть)
3. **JobStatusPanel** (если активный job есть)
   - статусная строка + короткий текст
   - действия: “повторить” (только если error и повтор доступен)
4. **VoiceRecorder** (нижняя панель)
   - Primary CTA: “Записать аудио послание”
   - после записи: кнопка “Отправить” (если так выбран flow)

## 4) VoiceRecorder — MVP flow (без стриминга)
### 4.1 State machine (клиент)
- `idle`
- `recording`
- `review` (preview после остановки записи)
- `uploading` (аудио загружается)
- `processing` (ожидаем transcript + assistantText + voiceReplyUrl via polling)
- `error`

### 4.2 UX-правила
- Запись и отправка — **одна кнопка в idle**, после записи показываем “Отправить”.
- Пока jobStatus != answered:
  - не даём повторно отправлять то же сообщение
  - но можно позволить пользователю начать новый чат (если он в другом chatId) — на MVP можно ограничить.

## 5) JobStatusPanel — отображение jobStatus
### 5.1 Статусы
- `ingested`
- `transcribing`
- `ready`
- `answered`
- `error`

### 5.2 Правила текста (без цифр, коротко)
- ingested: “Получил аудио. Обрабатываю…”
- transcribing: “Распознаю речь…”
- ready: “Смысл готов. Формирую ответ…”
- answered: “Ответ готов.”
- error: “Ошибка обработки. Попробовать ещё раз?”

## 6) Планы / cron / проекты — минимальная навигация
> MVP допускает упрощённые формы: главное — навигация, сущности и привязки.

### 6.1 Страница “Проекты”
- список проектов
- создание проекта (минимум: название)
- выбор “текущий проект”

### 6.2 Страница “Планы” (UI‑слой над cron)
- список планов
- “создать план” → связываем с проектом и cron_task

### 6.3 Страница “Cron”
- список cron_task
- кнопка: “Запуск сейчас” (опционально)
- кнопка: “Пауза/активация”

## 7) Поллинг-логика (UI-side, без стриминга)
### 7.1 Модель
- После `POST /api/chats/{chatId}/voice` получаем `jobId` и `messageId`.
- UI делает polling `GET /api/jobs/{jobId}` до `answered` или `error`.
- Период polling: 2–3 секунды (MVP), затем можно оптимизировать.

### 7.2 Отображение прогресса
- Параметр `progress` если есть — используем для тонкой полоски/лейбла.
- Если progress нет — просто меняем jobStatus текстом.

## 8) Критерии MVP по UI (приёмка)
- Переключение между чатами работает.
- Кнопка записи аудио доступна всегда в чате.
- После отправки появляется понятный статус.
- Transcript и assistantText отображаются после завершения обработки.
- Есть кнопка play для голосового ответа.
- Ошибки отображаются с понятным “повторить”.
